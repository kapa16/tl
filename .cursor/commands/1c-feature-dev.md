# /1c-feature-dev — полный цикл доработки 1С

Запусти скилл **`1c-feature-dev`** (`.cursor/skills/1c-feature-dev/SKILL.md`) и пройди фазы 0–8 для запроса пользователя.

Аргумент команды (текст после `/1c-feature-dev`) — исходная постановка задачи; передай его в Phase 0–1 как `$ARGUMENTS`.

## Перед стартом

1. Прочитай **`USER-RULES.md`** (политика проекта: расширение, префикс, каталог исходников).
2. При необходимости вызови **`recall`** (MCP `1c-templates-mcp`) по ключевым словам задачи.
3. Соблюдай **`AGENTS.md`**: MCP (`mcp-1c-tools`), валидация BSL, on-demand-правила по типу задачи.

## Субагенты и скиллы (канон для этого репозитория)

Не используй устаревшие имена `1c-code-*` из оригинального workflow — в проекте действует маппинг из `USER-RULES.md` → раздел «Feature-dev workflow».

| Фаза | Делегирование |
|------|----------------|
| 2 — исследование | `1c-explorer` |
| 4–5 — план / ревью плана | `1c-architect` или `1c-planner`; опционально `1c-arch-reviewer` |
| 6 — реализация | `1c-developer`; метаданные — `1c-metadata-manager` или скиллы `meta-*` / `cfe-*` / `form-*` |
| 6 — после этапа | `syntaxcheck` → `check_1c_code` → `review_1c_code` (см. `verification-checklist`) |
| 7 — ревью | соответствие плану — `1c-architect`; качество — MCP; `1c-code-reviewer` — **только если пользователь явно просит ревью** |
| 8 — приёмка | по запросу: `/deploy-and-test` или `1c-tester` + `web-test` |

## Альтернативные входы

- Идея без готового ТЗ: сначала скилл **`brainstorm`**, затем `/1c-feature-dev` с утверждённым дизайном.
- Крупная фича с OpenSpec: `/opsx:propose` или `/opsx:apply` **параллельно** с feature-dev (артефакты в `openspec/`, этапы — в `.tasks/`).

## Артефакты

Каталог: `.tasks/task-<краткое-имя>/` (`phase0-complexity.md` … `phase8-summary.md`).

Не переходи к Phase 6 без явного одобрения плана пользователем.
