# План упрощения разбора данных VLM и заполнения ведомости

Документ для будущей реализации (ИИ / разработчик).  
Источник анализа: ревью `src/epf/РаспознаваниеИИ/Распознавание ИИ/Ext/ObjectModule.bsl` (~3900 строк) и сравнение с `src/epf/ЗагрузкаЗаправочныхВедомостей`.

**Дата фиксации:** 2026-06-18.

---

## 1. Контекст и цель

### 1.1. Текущий поток данных

```
Скан (присоединённый файл)
  → VLM (одно- или двухпроходно)
  → [опционально] постобработка LLM
  → разбор JSON (repair + преобразование в Структуру)
  → нормализация схемы (header / rows / footer)
  → эвристики исправления ошибок VLM
  → заполнение Документ.GDS_ЗаправочнаяВедомость
  → визуальный контроль (форма EPF)
```

### 1.2. Публичный API обработки (не ломать без миграции)

| Метод | Назначение |
|-------|------------|
| `РаспознатьВедомость` | Только распознавание по файлу ведомости |
| `РаспознатьИЗаполнитьВедомость` | Распознавание + заполнение + запись |
| `ЗаполнитьВедомостьПоДаннымVLM` | Заполнение по уже готовым данным |
| `ЗапросРазспознаванияИИПоДвоичнымДанным` | Низкоуровневый вызов VLM |
| `СформироватьТаблицуВизуальногоКонтроля` | UI проверки |
| `СформироватьСводкуВизуальногоКонтроля` | UI сводки |
| `ПроверитьРаспознаваниеНаЭталоне` | Регресс на `ведомости.6_1` |

### 1.3. Каноническая схема JSON (version 2)

Эталон: `scans/результат_ведомости.6_1.json`.

Корень: `version`, `templateType` (`zapravka` \| `prihodnaya` \| `perelivnaya`), `header`, `rows`, `footer`, `warnings`, `errors`.

Каждое поле — объект `{ id, label, value, confidence, status, source }` (допускается `valueString`).

Строка таблицы: `{ rowIndex, isEmpty?, fields: { ... } }`.

### 1.4. Главный вывод анализа

| Блок | ~Строк | Вердикт |
|------|--------|---------|
| Заполнение ведомости (`#Область ЗаполнениеВедомости`) | ~600 | **Уже простой**, менять логику минимально |
| Разбор JSON + санитизация имён | ~650 | **Основной резерв** упрощения |
| Нормализация схемы | ~400 | Частично избыточен при стабильной постобработке |
| Эвристики `ПроверитьИИсправитьОтветVLM` | ~300 | Страховка; убирать только с регрессией |
| Промпты VLM | ~380 | Вынести в отдельный модуль (читаемость) |
| HTTP / VLM / ориентация | ~900 | Инфраструктура, не трогать без нужды |
| Визуальный контроль | ~350 | Дублирует заполнение при отсутствии связей |

**Дублирование:** `ЗагрузкаЗаправочныхВедомостей` (EPF и CF) содержит почти идентичные `ЗначениеПоляOCR` / `ЗаполнитьДокументПоOCR` / `НайтиИлиДобавитьСтрокуЗаправки` — общая схема полей с VLM.

## 2. Ограничения и риски (обязательно учитывать при реализации)

1. **Имена свойств Структуры** — VLM возвращает ключи вроде `Гос_№`, `кол-во ДТ (литр)`. Прямой `Структура.Свойство("Гос_№")` в BSL недопустим. Сейчас это решается цепочкой `АлиасыИменПолейJSONVLM` → `СанитизироватьИмяСвойстваСтруктуры` → `ИмяКлючаДоступаКСекции`.

2. **Fallback без постобработки** — в `ЗапросРазспознаванияИИПоДвоичнымДанным` при обрезанном ответе (`finish_reason = length`) или сбое LLM используется сырой текст VLM → нужен repair JSON.

3. **Эвристики** компенсируют реальные ошибки модели (сдвиг колонок литры↔температура, эталонная полоса `1234567890`, путаница `fuel_name`/`field_name`, копирование литров в кг). Удаление без тестов ухудшит качество на «грязных» сканах.

4. **Расширение `src/cfe/`** — при выносе в общий модуль CF версию расширения не трогать (правило versioning только для CFE). Изменения — в `src/cf/` и `src/epf/`.

5. **Регламентное задание** `GDS_ОбработатьВходящиеПисьмаРаспознать` сейчас использует `GDS_ЗагрузкаЗаправочныхВедомостей` (QR + OCR-утилита), **не** EPF `РаспознаваниеИИ`. Интеграция VLM в продакшен-поток писем — отдельное решение (этап 5).

### Этап 3. Хранение данных VLM в Соответствии (упрощение разбора JSON)

**Цель:** убрать ~150–200 строк санитизации имён для `Структура`.

**Предпосылка:** этапы 1–2 завершены; аксессоры полей работают через `ИмяКлючаДоступаКСекции` / алиасы на уровне **нормализации**, а не при каждом `ПрочитатьJSON`.

**Задачи:**

- [ ] Ввести тип данных «дерево VLM» как `Соответствие` (внутренне), API — по-прежнему может отдавать `Структуру` наружу для совместимости.
- [ ] `ПопытатьсяПрочитатьJSONВСоответствие` + `ПодготовитьТекстJSON` — основной путь; `ПреобразоватьJSONВСтруктуру` — только для legacy или удалить после миграции.
- [ ] Упростить `РазобратьJSONСтроку`: меньше вариантов текста, если постобработка стабильна (см. этап 4).
- [ ] Алиасы русских имён применять в `НормализоватьСтрокиТаблицы` / `ПрименитьАлиасыПолейСтроки`, не в `ИмяКлючаJSONДляСтруктуры`.

**Критерий готовности:** эталон зелёный; JSON с русскими ключами из промежуточных ответов (`scans/1`…`5`) разбирается без потери полей.

### Этап 4. Упрощение нормализации (при стабильной постобработке LLM)

**Цель:** сократить код, дублирующий работу LLM-постобработки.

**Предпосылки (подтвердить на реальных сканах):**

- в продакшене `БезПостобработки = Ложь`;
- постобработка не обрезается (`ОтветИИОбрезан` = Ложь) в >95% случаев;
- ответ уже в канонической вложенной схеме.

**Задачи (по приоритету, каждый — отдельный коммит с регрессией):**

- [ ] **4a.** `ПривестиПлоскуюСхемуККанонической` — оставить только как fallback (лог + флаг в warnings).
- [ ] **4b.** `ПривестиПоляСтрокиИзМассиваВСтруктуру` — удалить, если массив `fields` больше не встречается в `scans/промежуточные ответы моделей1.json` и новых логах.
- [ ] **4c.** Сократить `АлиасыИменПолейJSONVLM` — перенести редкие алиасы в промпт постобработки (`ПромптПостобработкиСистемный`).
- [ ] **4d.** Рассмотреть `response_format: json_object` или JSON Schema в теле запроса к API (если endpoint поддерживает) — уменьшит `ИзвлечьТелоJSONИзТекста` / markdown-strip.
- [ ] **4e.** Двухпроходное распознавание — оставить опцией; основной путь — однопроход + постобработка (упростить `ОбъединитьРезультатыПроходов` / `НуженПовторныйЗапросТаблицы`).

**Критерий готовности:** эталон + N реальных сканов; сравнение `ДанныеJSON` до/после на уровне заполненного документа (не обязательно побайтовое совпадение JSON).

**Риск:** средний–высокий. **Не объединять 4a–4e в один PR.**

| `ИсправитьКопированиеЛитровВКилограммы` | л↔кг в строках | Сохранить |

**Критерий готовности:** удалены только эвристики с доказательством ненужности; остальные задокументированы в комментарии к `ПроверитьИИсправитьОтветVLM`.

### Этап 6. Визуальный контроль

**Цель:** убрать дублирование двух путей заполнения таблицы контроля.

- иначе — `ЗаполнитьТаблицуВизуальногоКонтроляПоСвязям` vs `ЗаполнитьТаблицуВизуальногоКонтроляИзVLM`.

**Задачи:**

- [ ] Всегда передавать `СвязиСтрокVLM` из `ЗаполнитьВедомостьПоДаннымVLM` (уже делается при `ЗаполнятьВедомость`).
- [ ] Для режима «только распознавание» — один вызов `СмоделироватьЗаполнениеВедомостиДляКонтроля`, убрать `ЗаполнитьТаблицуВизуальногоКонтроляИзVLM` если логика эквивалентна.
- [ ] `СформироватьПредупреждениеЗаполненияСтроки` — общая с заполнением документа.

**Критерий готовности:** форма EPF показывает ту же таблицу до/после.

**Риск:** низкий.

## 5. Верификация (для каждого этапа)

1. `syntaxcheck` — все изменённые BSL-модули.
2. `ПроверитьРаспознаваниеНаЭталоне()` — без новых ошибок.
3. Ручной прогон EPF: ведомость + скан, флаги `ЗаполнятьВедомость` / `ДвухпроходноеРаспознавание` / `БезПостобработки`.
4. Сравнение таблицы визуального контроля и `КоличествоВсего` до/после рефакторинга на одном и том же JSON.
5. Для этапа 2 — прогон `ЗагрузкаЗаправочныхВедомостей` с OCR-JSON (без VLM).

## 6. Что НЕ делать

- Не удалять эвристики пакетом без эталона и реальных сканов.
- Не менять публичные сигнатуры EPF без явной задачи миграции.
- Не смешивать этап 7 (письма) с этапами 1–3 в одном PR.
- Не заменять кастомный repair JSON только на `ОбщегоНазначения.JSONВЗначение` без `ПодготовитьТекстJSON` (markdown, trailing commas).
- Не хранить русские ключи JSON в `Структуре` без алиасов или `Соответствия`.
