# Phase 2: Исследование кодовой базы

## Стратегия

Один агент `1c-explorer` с фокусом на EPF, документ, печать, сканы. MCP graph/code-metadata в сессии недоступны — факты из исходников и grep.

## Найденные паттерны

### EPF — внешние утилиты

```
ДвоичныеДанные → [PDF?] ImageMagick → PNG → temp-файл
→ exe из макета → ФайловаяСистема.ЗапуститьПрограмму (UTF-8 stdout)
→ разбор JSON → запись документа
```

Файл: `src/epf/ЗагрузкаЗаправочныхВедомостей/.../Ext/ObjectModule.bsl`

### Документ GDS_ЗаправочнаяВедомость

- Шапка: `Дата`, `КоличествоВсего`, `ТипВедомости`, `АЗС`, `ГСМ`, `Основание`
- ТЧ `Заправки`: `Количество`, `Плотность`, `ДатаЗаправки`, `НомерПЛ`, `ТС`, `ПЛ`, …
- Проведение требует `АЗС`, `ГСМ`, `ТС`, `ПЛ` — OCR количеств ≠ готовность к проведению

### Перечисление типов

`Приходная`, `Заправочная`, `Переливная`, `Инвентаризационная` — **нет «Передвижная»** (бланк `.1.jpg`)

### Печать

- `GDS_ПечатьЗапровочныхВедомостей` — один MXL `Ведомость` (соответствует `.3.jpg`)
- QR в печати: `АЗС`, `Контрагент`, `ТипВедомости` (не `ВедомостьУИН`)
- EPF ожидает QR с `ВедомостьУИН` — **расхождение**

### Предыдущая задача `task-email-qr-fuel-statement`

- EPF, gozxing, ImageMagick, БСП `РаботаСФайлами`
- Документ непроведённый, `ОбменДанными.Загрузка`
- Исходники `tools/qrcoderecognizer` в репо **отсутствуют**

## Архитектурные инсайты

1. OCR-утилита — отдельный exe в макете EPF (`КомпонентаOCR`), CLI → JSON
2. Маски областей — JSON по типу ведомости (координаты из img-grid + калибровка)
3. Определение типа бланка — из QR `ТипВедомости` или эвристика по заголовку
4. Заполнение — после `ВедомостьПоДаннымQR`, до `Записать()`
5. Образец `1234567890` — template matching или обучение на эталоне с бланка

## Риски

- 3 бланка / 1 MXL в CF
- «Передвижная» не в enum
- QR print ≠ QR EPF
