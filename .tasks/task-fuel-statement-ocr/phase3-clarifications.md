## Ответы пользователя (2026-06-15)

| # | Вопрос | Ответ |
|---|--------|-------|
| 1 | Создание vs поиск документа | **Как сейчас:** найти существующую ведомость по `ВедомостьУИН` из QR, заполнить OCR-поля |
| 2 | Формат QR | Не менять в рамках задачи (оставить `ВедомостьУИН`) |
| 3 | Бланк `ведомости.1.jpg` | Это **`Переливная`** (не «Передвижная») |
| 4 | Объём OCR | **Все колонки с гребёнками** (дата, литры, кг, гаражный №, плотность и т.д. по бланку) |
| 5 | Технология | **Go + OpenCV**, template matching по образцу `1234567890`, exe в макете EPF (как gozxing) |
| 6 | Ошибки OCR | **Частичная запись** + комментарий/лог |
| 7 | UI | Не выбрано явно → **лог в форме EPF** (без предпросмотра в v1, по аналогии с QR-обработкой) |

## Принятые решения

1. OCR — этап **после** успешного QR и получения ссылки на документ, **до** `Записать()`.
2. Тип бланка для маски — из реквизита `ТипВедомости` документа (уже известен после поиска по УИН).
3. Соответствие сканов:
   - `ведомости.3.jpg` → `Заправочная`
   - `ведомости.2.jpg` → `Приходная`
   - `ведомости.1.jpg` → `Переливная`
4. Новая утилита `FuelStatementOCR` (Go) + макет `КомпонентаOCR` в EPF.
5. Маски областей — JSON в репозитории (`tools/fuel-statement-ocr/templates/`), встроены в утилиту или рядом.
6. При частичном OCR — записать распознанное, в `Комментарий` документа и лог EPF — список нераспознанных ячеек.
