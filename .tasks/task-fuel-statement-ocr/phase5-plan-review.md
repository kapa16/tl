## Critical Issues (Must Fix)

### 1. Не учтена ориентация эталонных сканов (поворот 90°) (Confidence: 88%)

**Category:** Maintainability / Performance (корректность CV)  
**Location:** §2 Go-утилита (preprocess/anchor), §4 JSON template (`orientation: "portrait"`), Этап 2–3

**Issue:** Эталоны `ведомости.2.jpg` и `ведомости.3.jpg` — **ландшафтные бланки, отсканированные/сфотографированные с поворотом ~90°** (контент повёрнут относительно краёв изображения). План описывает deskew и аффинное выравнивание по якорям, но **не предусматривает явное определение и нормализацию ориентации** (90/180/270°) до применения нормализованных координат масок.

**Why It Matters:** Координаты `x/y/w/h` в JSON привязаны к `referenceSize` в «канонической» ориентации. Без шага `auto-rotate` якоря QR и полоса `1234567890` не попадут в ожидаемые ROI → `ANCHOR_NOT_FOUND` на реальных вложениях, даже при точной калибровке на «неповёрнутых» crop.

**Evidence:** Визуальный осмотр `scans/ведомости.{2,3}.jpg`; в phase2 зафиксирован риск «наклон скана», но не rotation.

**Recommended Fix:**
- Добавить в preprocess этап **EXIF orientation** + эвристику по соотношению сторон и/или положению QR/полосы цифр.
- В JSON-шаблоне: `canonicalOrientation`, `allowedRotations`, опционально `contentRotationDegrees` для эталона.
- В Этап 2: калибровка **после** приведения к канонической ориентации; debug `--dump-crops` на исходном и нормализованном изображении.
- Критерий приёмки Этапа 2: ≥90% crop на **исходных** файлах из `scans/`, без ручного поворота.

**Trade-offs:** Увеличивает сложность preprocess; зато без этого остальная архитектура (якоря + доли координат) не работает на двух из трёх бланков.

## High Issues (Should Fix)

### 2. Противоречие: фатальная ошибка OCR vs «частичная запись» (Confidence: 82%)

**Category:** Maintainability / соответствие требованиям  
**Location:** §3 JSON (`anchorsFound: false` → exit ≠ 0), §5 политика записи, phase3 п.6

**Issue:** Пользователь требует **частичную запись + комментарий/лог** при ошибках OCR. План одновременно задаёт **ненулевой код возврата** при `ANCHOR_NOT_FOUND` и не описывает, что BSL делает в этом случае: прерывает ли весь `ОбработатьПисьмо` (как при исключении) или сохраняет документ по QR без OCR-полей.

**Why It Matters:** Текущий `ВедомостьПоДаннымQR` в транзакции пишет документ и файл. При вставке OCR **между** подготовкой и `Записать()` фатальный OCR легко превратится в полный отказ письма — это **противоречит** phase3.

**Recommended Fix:** Явная политика в §2 и Этап 5:
- **OCR — best-effort:** при exit ≠ 0 или `anchorsFound = false` → не менять количественные поля, дописать `Комментарий` + `ЛогOCR`, **всё равно** `Записать()` + вложение (как сейчас по QR).
- Ненулевой exit — только для «не читается файл / неизвестный type / битый JSON».
- Обновить sequence diagram: ветка «OCR failed → save QR-only».

**Trade-offs:** Оператору нужен явный статус «без OCR» в форме (Этап 6).

### 3. Эталон `ведомости.3.jpg` без рукописных данных в гребёнках (Confidence: 80%)

**Category:** Реалистичность критериев приёмки  
**Location:** Этап 3, Этап 7 (Заправочная)

**Issue:** На эталоне `.3.jpg` (ДЕЛЬТА-ОЙЛ, Заправочная) видны полоса `1234567890` и структура таблицы, но **нет заполненных рукописных цифр в строках** (в отличие от `.1.jpg` с 123/456/79). Критерии Этапа 7 требуют «≥1 строка `Количество`» на этом скане — **невыполнимо** без синтетических данных или другого образца.

**Recommended Fix:**
- Зафиксировать в плане **набор приёмочных сканов**: эталоны структуры (`scans/`) + минимум один **заполненный** скан на тип (или синтетическая генерация в `tools/.../testdata/`).
- Ослабить/уточнить критерий для `.3.jpg`: «корректные crop + распознавание полосы цифр; строки — на testdata или ручной доработке скана».
- Добавить подзадачу в Этап 2: подготовить/запросить у заказчика заполненный образец Заправочной.

### 4. Маппинг `rowIndex` → ТЧ `Заправки` недоопределён (Confidence: 78%)

**Category:** Maintainability / целостность данных  
**Location:** §5 «Алгоритм построения строк», п.3

**Issue:** Документ по УИН **уже может содержать строки ТЧ**, созданные в 1С до загрузки скана. План: «дополнять по номеру строки бланка если строки уже есть; иначе `Добавить()`» — без правил:
- сопоставление по `НомерСтроки` / позиции / пустым строкам;
- поведение при **большем** числе строк в документе, чем на бланке;
- перезапись `Количество` при `status=ok` vs уже заполненное значение.

**Why It Matters:** Риск дублирования строк, записи литров не в ту ТС/строку, расхождения с данными печатной формы.

**Recommended Fix:** Добавить таблицу решений:
- v1: **строка бланка N → строка ТЧ с тем же номером**, если существует; иначе новая строка в конце с сохранением `rowIndex` в комментарии OCR;
- не трогать строки без данных на бланке;
- при `low_confidence` — не перезаписывать непустой реквизит (уже в псевдокоде — вынести в нормативное правило).

### 5. Год даты документа при OCR дня/месяца (Confidence: 76%)

**Category:** Корректность бизнес-логики  
**Location:** §5 шапка, `date_day` + `date_month`

**Issue:** Правило: год из **текущей `Дата` документа** (сейчас подставляется `ДатаПолучения` письма). На бланке год **напечатан** (`2026`). Если документ создан заранее с другой датой или письмо пришло в следующем году — OCR даст неверную дату.

**Recommended Fix:** Приоритет: (1) напечатанный год с бланка, если добавить поле `date_year` в маску/JSON; (2) иначе год из `Дата` документа; (3) иначе год из `ДатаПолучения`. Зафиксировать в контракте JSON и маппинге.

---

### 6. Нет проверки соответствия `ТипВедомости` и фактического бланка (Confidence: 75%)

**Category:** Robustness  
**Location:** §2 поток данных, `ИмяТипаДляOCR(ТипВедомости)`

**Issue:** Маска выбирается **только** из реквизита документа. Ошибка типа при создании документа / неверное вложение → неверная маска → мусор в полях или тихий partial.

**Recommended Fix:** v1-минимум: после выравнивания — **эвристика заголовка** (ключевые слова «Приходная» / «Заправочная» / «Ведомость учета ГСМ») → `warning` при расхождении с `--type`, не блокировать запись. Опционально: понизить confidence всех полей при mismatch.

---

### 7. Зависимость OpenCV: «один exe» может быть нереалистична (Confidence: 72%)

**Category:** Реалистичность поставки  
**Location:** §7 риски, структура Go

**Issue:** Для Go + OpenCV (gocv) на Windows часто требуются **DLL OpenCV**, а не только статический exe. План ссылается на статическую линковку без этапа проверки размера/зависимостей макета `КомпонентаOCR`.

**Recommended Fix:** В Этап 1/4 добавить критерий: «на чистой ВМ 1С запускается **только** exe из макета без PATH»; зафиксировать стратегию (static build / bundle DLL рядом с exe / отдельный макет). Оценить лимит размера макета EPF.

### 9. Политика `partial` vs запись значений (Confidence: 70%)

**Location:** §5 псевдокод

**Issue:** Поле со `status=partial` записывается, ячейки с `low_confidence` — нет. Не определено, как собирать `value` при частично пустых ячейках гребёнки (например, `1?3` литра).

**Fix:** Правило сборки: `partial` → записывать `value` только если все **значащие** ячейки ≥ порога; иначе только `valueString` в комментарий.

### 10. `location_code` → `Локация` (шапка) (Confidence: 68%)

**Location:** §5 маппинг

**Issue:** На бланке Переливной колонка «Локация» — скорее **текстовая**, не гребёнка. План допускает «если гребёнка» — хорошо, но в сводке §4 колонка указана без уточнения. Риск ложной маски.

**Fix:** Для `perelivnaya.json` явно исключить `location_code` из v1 или пометить `optional: false` после проверки img-grid.

### 11. Этап 4 жёстко после Этапа 3 (Confidence: 65%)

**Issue:** BSL-обёртку и мок JSON можно делать параллельно с Этапом 2–3 (ранняя интеграционная проверка).

**Fix:** Разрешить Этап 4 с заглушкой OCR сразу после Этапа 1; переключить на реальный exe после Этапа 3.

### 13. Расширение `ОбработатьПисьмо` не спланировано в одном месте (Confidence: 62%)

**Зависимости между этапами:** логичны (1→2→3→4→5→6→7). Рекомендуется **размягчить 4↔3** (п.11).

**Внешние зависимости:** ImageMagick (PDF), OpenCV runtime (п.7), БСП `РаботаСФайлами` — отражены.

**Дополнительный техдолг (рекомендуется добавить в план):**

- Заполненные приёмочные сканы / testdata
- Стратегия поставки OpenCV на сервер 1С
